# Skills Hub - 产品需求规格 (Spec v0.3)

> 内部部署的 **下载式** AI 技能共享平台：让企业的 AI Skill 与数字员工可被发现、可被治理、可被下载安装到员工自己常用的 AI 工具中。

> 修订历史：
>
> - **v0.2** Review 修订（M1~~M5 + S1~~S11 + 全部补充项）
> - **v0.2.1** 首个 Owner 凭据从 `.env` 读取，seed 脚本不写死默认凭据
> - **v0.3** 架构 pivot：Skill 改为**下载式静态包**（`SKILL.md` + 资源文件），移除 HTTP 调用 / API Key / CallLog / 试运行 / OpenAPI 渲染 / 异步任务。完整背景见 [`decisions/adr-001-download-style-skill-hub.md`](./decisions/adr-001-download-style-skill-hub.md)
>
> 完整变更见文末"修订记录"。

## 1. 产品概述

### 1.1 一句话定义

Skills Hub 是公司内部的 **AI 技能与数字员工分发市场**：把分散在各工具链、Wiki、IM 机器人中的 AI Skill 与"带人设的数字员工"集中沉淀到统一仓库，让员工可发现、可下载、可在自己的 AI 工具里安装使用。

### 1.2 业务背景

- 公司目前拥有大量 AI Skill 与数字员工，但分散在各个工具链、Wiki、IM 机器人中
- 员工不知道有哪些可用能力，找不到、用不上、用不好
- 技能质量参差，缺乏统一审核、版本管理与分发规范
- Skill 实际使用形态主要是 `SKILL.md` + 资源文件，由员工在自己常用的 AI 工具（OpenCode / Claude Desktop / Cursor / Codex 等）中加载——**不是常驻 HTTP 服务**

### 1.3 产品目标

| 目标       | 度量                                            | 基线                |
| ---------- | ----------------------------------------------- | ------------------- |
| 能力中心化 | 90 天内全部高频 Skill 与数字员工完成入库        | 高频 = 月下载 > 0   |
| 可发现     | 技能发现路径 ≤ 3 次点击                         | 从首页到达详情      |
| 可分发     | 每个 Skill 都能一键下载并附带清晰的安装指引     | 安装指引覆盖率 100% |
| 可治理     | 任何 Skill 都有归属人、版本、审核状态、下载数量 | 100% 覆盖           |

### 1.4 范围 vs 非范围

**In Scope (MVP)**

- Skill 与数字员工的发布、分类、搜索、详情
- 上下架与审核流程（4 角色）
- **zip 包上传 / 下载 / 多 AI 工具安装指引**
- 用户反馈（点赞 / 收藏 / 评分 / 评论）
- 下载量 / 评分 / 收藏数等站内可观测统计
- 账号体系：自建账号注册/登录；首个超级管理员（Owner）**通过 seed 脚本从 `.env` 读取凭据（`SEED_OWNER_EMAIL/SEED_OWNER_NAME/SEED_OWNER_PASSWORD`）创建，不在代码/数据库中硬编码**；后续用户由 Owner 邀请并分配角色

**Out of Scope (后续版本)**

- 多租户/多 BU 隔离
- 付费/计费
- SSO（企业微信/飞书/OIDC）—— v2 接入
- 跨 Hub 的 Skill 同步 / 联邦
- **在线试运行 / HTTP 调用 / API 化开放** —— 长期方向保留，若需求强烈，写新 ADR 引入
- 异步任务与长任务调度

> **v0.3 关键边界**：Hub 只做发现 / 治理 / 分发，**不做调用**。Skill 的执行发生在用户本地的 AI 工具中，Hub 不承担模型层 / 推理网关角色。

### 1.5 术语表

| 术语             | 定义                                              |
| ---------------- | ------------------------------------------------- |
| Skill            | 最小复用单元，一个目录：`SKILL.md` + 可选资源文件 |
| Digital Employee | 带人设的 Skill（= Skill + 人设层）                |
| 贡献者           | 可发布 Skill 的角色                               |
| 审核员           | 可审核待上架 Skill 的角色                         |
| Skill 包         | Skill 目录打成的 zip，平台唯一分发格式            |
| 安装指引         | 平台自动渲染的、针对各 AI 工具的目录放置说明      |

## 2. 用户与角色

采用经典四角色模型：

| 角色                         | 关键能力                                                                                                           | 典型场景                                                 |
| ---------------------------- | ------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------- |
| **普通用户 (Visitor)**       | 浏览、搜索、下载 Skill 包、按指引安装、收藏、评分、评论、点赞                                                      | 找一个能抽取合同条款的 Skill，下载到 Claude Desktop 里用 |
| **技能贡献者 (Contributor)** | 在普通用户能力之上：创建/编辑自己的 Skill 与数字员工、上传 zip、提交审核、查看自己 Skill 的下载/反馈数据、自主下架 | 把自己写的"合同条款抽取" Skill 发布                      |
| **技能审核员 (Reviewer)**    | 在贡献者能力之上：进入审核队列、通过/驳回、打回修改、填写审核意见                                                  | 检查新提交 Skill 的安全性与质量                          |
| **超级管理员 (Hub Owner)**   | 全量能力：用户与角色管理、分类管理、全局配置、审计日志、**强制下架**任意 Skill、数据看板                           | 给团队开通账号、定义分类、查分发大盘、紧急下架违规 Skill |

> 数字员工（Digital Employee）是**带人设的高级 Skill**——拥有名字、头像、人设描述、欢迎语，但本质是 Skill 的一种特殊形态，复用同一套发布/下载/审核流程。

> **重要边界**：**强制下架**是高危操作，仅 Hub Owner 拥有该权限；Reviewer 只能"打回修改 / 驳回"，不能直接下架已上架的 Skill。

## 3. 核心概念

### 3.1 Skill（技能）

- 最小复用单元，**一个目录**，包含：
  - `SKILL.md`（必须）：技能指令，YAML frontmatter + Markdown 正文
  - `agents/openai.yaml`（推荐）：UI 展示用元数据
  - `scripts/`、`references/`、`assets/`（可选）：脚本、参考文档、资源
- 包含元数据：slug、名称、归属人、分类、标签、版本号、审核状态
- **分发形态**：zip 包，平台存储并提供下载；用户解压后放到对应 AI 工具的 skills 目录

### 3.2 SKILL.md 与 Schema 规范

- **SKILL.md 格式**：采用 WorkBuddy 的 `SKILL.md` 规范，含 YAML frontmatter（name/description/triggers 等）+ Markdown 正文
- **运行期校验**：Skill 内部若声明入参契约（在 frontmatter 或正文里），由用户本地的 AI 工具自行解析与校验，**平台不强制、不解析、不校验**——平台只把 SKILL.md 当作 Markdown 内容存储与展示
- 平台不渲染 OpenAPI 片段，不生成 cURL 示例

### 3.3 Digital Employee（数字员工）

- **= 一个 Skill + 人设层**
- 在 Skill 基础上额外拥有：名称、头像、角色定位、欢迎语、典型场景
- 在 UI 中以"人"的卡片形式呈现，但本质仍走同一套下载/审核流程
- 头像统一存于对象存储（S3/MinIO），不允许外链引用
- 可理解为"被包装好的 Skill"

### 3.4 Skill 执行模型（v1：下载式）

v1 阶段 Skill 是**静态指令包**：

- 平台**不调用** Skill，**不**提供模型层 / 推理网关
- 用户在详情页下载 zip → 解压 → 按平台提供的安装指引放到对应 AI 工具的 skills 目录：
  - OpenCode / opencode：`~/.workbuddy/skills/<skill>/`
  - Claude Desktop / Claude.ai：`~/.claude/skills/<skill>/`
  - Cursor / VS Code：项目内 `.cursor/skills/` 或 `.vscode/skills/`
  - OpenAI Codex / ChatGPT：上传 SKILL.md 或粘贴到 Custom Instructions
- Skill 的实际执行（推理、工具调用）由员工本地的 AI 工具承担
- 平台无法、也无需采集真实"调用次数"——统计指标改用下载量、收藏数、评分等站内可观测数据

> 若未来确实需要"在 Hub 内试运行 Skill"或"通过 HTTP API 调用 Skill"，应新开 ADR 评估，引入沙箱 / 网关 / API Key 体系。

## 4. 功能清单

### 4.1 浏览与发现

- **F1 首页** —— 数字员工与 Skill 入口分流、推荐位、最新上架、热门下载
- **F2 列表页** —— 多维筛选（分类/标签/状态/更新时间）、关键词搜索、排序（最新/最热/评分）
- **F3 详情页** —— 数字员工展示人设卡片；Skill 展示 SKILL.md 渲染、版本、文件树、下载入口、安装指引、反馈
- **F4 分类管理** —— 树形分类，Hub Owner 可维护

### 4.2 发布与生命周期

- **F5 Skill 创建向导** —— 填写基础信息 → 编写/上传 SKILL.md → 上传 zip 资源包（可选）→ 提交审核
- **F6 数字员工创建** —— 在 Skill 基础上叠加人设（名字、头像、欢迎语、角色说明）
- **F7 版本管理** —— 每次发布生成版本号（SemVer），支持查看历史与回滚
- **F8 上下架** —— 贡献者可下架自己的（`POST /api/studio/skills/:id/offline` → 状态 Offline）；**Hub Owner 可强制下架任意 Skill（留原因）**；Reviewer 不能直接下架已上架 Skill，只能打回修改

### 4.3 审核流

- **F9 审核队列** —— Reviewer 视角查看待审 Skill
- **F10 审核操作** —— 通过 / 驳回（带意见）/ 打回修改
- **F11 审核历史** —— 每个 Skill 完整记录审核人、时间、意见

### 4.4 下载与安装

- **F12 Skill 包下载** —— 详情页提供 zip 下载入口；服务端从 SkillFile 表实时打包，含 SKILL.md + 所有资源文件；下载前可见文件大小
- **F13 安装指引自动渲染** —— 详情页根据 Skill 类型自动渲染安装步骤，覆盖主流 AI 工具（OpenCode / Claude / Cursor / VS Code / Codex），含目录路径与示例
- **F14 文件树预览** —— 详情页展示 Skill 包内文件结构（树形），用户下载前可预览包内容；点击单文件可在线查看文本内容（图片等二进制只展示元信息）

### 4.5 反馈与社区

- **F16 评分** —— 1-5 星
- **F17 评论** —— 富文本评论，支持**一层回复**（不嵌套）
- **F18 收藏** —— 我的收藏列表
- **F19 点赞** —— 详情页点赞计数

### 4.6 数据与统计

> v0.3 起，统计指标全部基于**站内可观测行为**（下载、收藏、评分、点赞），不再依赖调用日志。

- **F20 技能统计** —— 每个 Skill 详情页显示：累计下载量、平均评分、收藏数、点赞数（v1 只累计，不做时间窗；时间窗趋势 v1.1 引入 `DownloadEvent` 表后再补）
- **F21 贡献者统计** —— 我发布的、上架数、累计下载量
- **F22 Hub 大盘** —— Owner 视角：总 Skill 数、下载趋势、Top 10、活跃贡献者

### 4.7 管理后台

- **F23 用户与角色** —— 增删改查、**邀请注册**、角色分配
- **F24 分类管理** —— 树形增删改
- **F25 审计日志** —— 关键操作留痕（可检索、可导出）
- **F26 强制下架** —— Owner 可对任意 Skill 强制下架并留原因

## 5. 关键用户故事

> 格式：作为 [角色]，我希望 [能力]，以便 [业务价值]

### 5.1 业务用户

1. 作为 **普通用户**，我希望**用关键词搜索"合同条款抽取"**，以便**快速找到相关 Skill**
2. 作为 **普通用户**，我希望**在详情页预览 SKILL.md 与文件树**，以便**下载前判断这个 Skill 是否符合预期**
3. 作为 **普通用户**，我希望**一键下载 Skill zip 包并看到清晰的安装指引**，以便**不写代码也能把它装到我常用的 AI 工具里**
4. 作为 **普通用户**，我希望**对用过的 Skill 打分评论**，以便**帮助其他同事做选择**
5. 作为 **普通用户**，我希望**对评论进行回复**（一层），以便**与贡献者或其他用户交流细节**
6. 作为 **普通用户**，我希望**给 Skill 点赞**，以便**快速表达对优质 Skill 的认可**

### 5.2 贡献者

7. 作为 **技能贡献者**，我希望**用向导一步步创建 Skill 并上传 zip**，以便**降低发布门槛**
8. 作为 **技能贡献者**，我希望**管理自己的版本（SemVer）**，以便**修复问题不破坏已下载用户的版本选择**
9. 作为 **技能贡献者**，我希望**看到我发布 Skill 的下载量与反馈**，以便**评估其价值**
10. 作为 **技能贡献者**，我希望**自主下架自己的 Skill**，以便**主动下线不再维护的能力**

### 5.3 审核员

11. 作为 **技能审核员**，我希望**在队列中集中审核待审 Skill**，以便**高效完成审核工作**
12. 作为 **技能审核员**，我希望**对不达标的 Skill 给出驳回意见或打回修改**，以便**贡献者理解并修改**

### 5.4 管理员

13. 作为 **超级管理员**，我希望**邀请用户并分配角色**，以便**为团队开通合适权限**
14. 作为 **超级管理员**，我希望**查看全 Hub 下载大盘**，以便**了解 AI 能力分发全景**
15. 作为 **超级管理员**，我希望**对违规 Skill 强制下架**，以便**及时止损**
16. 作为 **超级管理员**，我希望**导出审计日志**，以便**满足合规审计需要**

## 6. 用户流程（关键路径）

### 6.1 Skill 提交流程

```
贡献者点击"发布 Skill"
  → 填写基础信息（名称/分类/标签）
  → 编写 SKILL.md（在线编辑器 + 实时预览，按 WorkBuddy SKILL.md 规范）
  → 上传 Skill zip 资源包（可选，仅 SKILL.md 也可）
  → 选择是否绑定为数字员工（是：补人设信息；否：跳过）
  → 提交审核
    → 状态：待审核 (Pending)
  → Reviewer 在审核队列处理
    → 通过 → 状态：已上架 (Approved)
    → 驳回 → 状态：已驳回 (Rejected)（带意见，贡献者修改后可重新提交）
    → 打回修改 → 状态：草稿 (Draft)（带意见，贡献者改完再次提交）
```

### 6.2 用户下载使用流程

```
普通用户进入详情页
  → 浏览 SKILL.md 渲染、查看文件树预览、看评分/评论
  → 点击"下载 Skill (.zip)"
  → 解压下载的 zip
  → 按详情页"如何安装"指引，把目录放到对应 AI 工具的 skills 目录
    （OpenCode: ~/.workbuddy/skills/；Claude: ~/.claude/skills/；Cursor / VS Code: 项目内）
  → 重启或重新加载 AI 工具，在对话中触发该 Skill
```

### 6.3 Skill 重新上架流程

```
贡献者 / Owner 将已下架的 Skill 重新发布
  → 进入我的工作台 → 选择 Offline 状态的 Skill
  → 点击"重新发布"
  → 可选：编辑新版 SKILL.md / 上传新 zip（生成新版本号，SemVer 自增 patch）
  → 提交审核
    → 状态：待审核 (Pending)
  → Reviewer 审核 → Approved（重新对普通用户可见，可下载）
```

### 6.4 强制下架流程（Owner）

```
Hub Owner 在管理后台/详情页点击"强制下架"
  → 弹窗填写下架原因（必填，10~500 字）
  → 确认 → 状态：已下架 (Offline)
  → 自动生成审计日志（actor/动作/原因/时间/目标）
  → 通知 Skill 归属人
  → 已下载用户本地副本不受影响（无法远程回收）
```

## 8. 风险与假设

### 8.1 假设

- 公司允许内部部署，使用自托管 PostgreSQL + S3 兼容对象存储（MinIO 或自有 S3）
- v1 阶段用自建账号，SSO 在 v2 接入
- **Skill 的执行发生在用户本地 AI 工具中**，Hub 不提供模型层、不承担推理网关角色
- 单 Owner 即可，v1 不要求多人 Owner
- **首个 Owner 凭据完全从 `.env` 读取**（email/name/password），seed 脚本不写死任何默认值；env 缺失时 seed 进程必须报错退出，避免静默创建弱口令账号

### 8.2 风险

| 风险                 | 等级 | 缓解                                                              |
| -------------------- | ---- | ----------------------------------------------------------------- |
| 状态机被绕过         | 高   | 全部状态转移集中在 `lib/skill/state.ts`，API 路由与 UI 都必须调用 |
| 越权访问             | 高   | `assertCan(user, action, resource)` 中间件强制每个 API 调用       |
| 强制下架被滥用       | 中   | 仅 Owner 可操作 + 必须填原因 + 必留审计                           |
| 上传 zip 含恶意脚本  | 中   | 服务端解压时校验路径穿越、文件大小、扩展名；Reviewer 人工兜底     |
| 用户下载后无法跑起来 | 中   | 安装指引覆盖主流工具 + 评分/评论机制反向暴露问题 Skill            |
| 用户数增长后搜索慢   | 低   | 1k 级别 PG 模糊搜索足够；1w+ 引入 ES                              |
| 自建账号被刷注册     | 中   | 注册需邀请码 / Owner 邀请激活（v1.1 增强）                        |

### 8.3 合规

- 审计日志需可导出为 CSV/JSON，保留期 ≥ 1 年
- 头像与 Skill zip 资源统一存对象存储，公开访问走签名 URL
- Skill 包内不得包含密钥、凭据、内部 IP 等敏感信息（审核员人工检查 + 关键词扫描兜底，v1.1 引入）
