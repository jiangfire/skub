> 内部部署的 AI 技能共享平台，让企业的 AI 能力、数字员工可被发现、可被调用、可被治理。

## 1. 产品概述

### 1.1 一句话定义

Skills Hub 是公司内部的 **AI 技能与数字员工市场**：把分散的 AI Skill 和"带人设的数字员工"集中沉淀到统一仓库，让员工可发现、可试运行、可通过 API 调用。

### 1.2 业务背景

- 公司目前拥有大量 AI Skill 与数字员工，但分散在各个工具链、Wiki、IM 机器人中
- 员工不知道有哪些可用能力，找不到、用不上、用不好
- 技能质量参差，缺乏统一审核、版本管理与调用规范
- 业务侧需要把 AI 能力以 API 形式稳定开放给内部系统

### 1.3 产品目标

| 目标       | 度量                                          | 基线              |
| ---------- | --------------------------------------------- | ----------------- |
| 能力中心化 | 90 天内全部高频 Skill 与数字员工完成入库      | 高频 = 月调用 > 0 |
| 可发现     | 技能发现路径 ≤ 3 次点击                       | 从首页到达详情    |
| 可调用     | 每个 Skill 配套稳定 API，可被其他系统集成     | API 可用率 ≥ 99%  |
| 可治理     | 任何 Skill 都有归属人、版本、审核状态、调用量 | 100% 覆盖         |

### 1.4 范围 vs 非范围

**In Scope (MVP)**

- Skill 与数字员工的发布、分类、搜索、详情
- 上下架与审核流程（4 角色）
- 在线试运行 + API 开放
- 调用统计与日志
- 用户反馈（点赞 / 收藏 / 评分 / 评论）
- **账号体系**：自建账号注册/登录；首个超级管理员（Owner）**通过 seed 脚本从 `.env` 读取凭据（`SEED_OWNER_EMAIL/SEED_OWNER_NAME/SEED_OWNER_PASSWORD`）创建，不在代码/数据库中硬编码**；后续用户由 Owner 邀请并分配角色

**Out of Scope (后续版本)**

- 多租户/多 BU 隔离
- 付费/计费
- SSO（企业微信/飞书/OIDC）—— v2 接入
- 跨 Hub 的 Skill 同步 / 联邦
- 异步任务与长任务调度 —— v1.1 引入

| 术语             | 定义                               |
| ---------------- | ---------------------------------- |
| Skill            | 最小复用单元，封装一段 AI 能力     |
| Digital Employee | 带人设的 Skill（= Skill + 人设层） |

## 2. 用户与角色

采用经典四角色模型：

| 角色                         | 关键能力                                                                                                        | 典型场景                                                 |
| ---------------------------- | --------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------- |
| **普通用户 (Visitor)**       | 浏览、搜索、试运行、收藏、评分、评论、点赞、调用 API                                                            | 找一个能抽取合同条款的 Skill                             |
| **技能贡献者 (Contributor)** | 在普通用户能力之上：创建/编辑自己的 Skill 与数字员工、提交审核、查看自己 Skill 的调用数据、自主下架自己的 Skill | 把自己写的"合同条款抽取" Skill 发布                      |
| **技能审核员 (Reviewer)**    | 在贡献者能力之上：进入审核队列、通过/驳回、打回修改、填写审核意见                                               | 检查新提交 Skill 的安全性与质量                          |
| **超级管理员 (Hub Owner)**   | 全量能力：用户与角色管理、分类管理、全局配置、审计日志、**强制下架**任意 Skill、数据看板                        | 给团队开通账号、定义分类、查调用大盘、紧急下架违规 Skill |

> 数字员工（Digital Employee）是**带人设的高级 Skill**——拥有名字、头像、人设描述、欢迎语，但本质是 Skill 的一种特殊形态，复用同一套发布/调用/审核流程。

> **重要边界**：**强制下架**是高危操作，仅 Hub Owner 拥有该权限；Reviewer 只能"打回修改 / 驳回"，不能直接下架已上架的 Skill。

## 3. 核心概念

### 3.1 Skill（技能）

- 最小复用单元，封装一段 AI 能力（提示词 + 工具 + 输入输出契约）
- 包含：`SKILL.md` 描述、版本号、输入 Schema、输出 Schema、归属人、分类、标签
- 调用形态：HTTP API（同步）

### 3.2 SKILL.md 与 Schema 规范

- **SKILL.md 格式**：采用 WorkBuddy 的 `SKILL.md` 规范，含 YAML frontmatter（name/description/triggers 等）+ Markdown 正文
- **输入输出 Schema**：采用 **JSON Schema**（与 OpenAPI 兼容），运行期用 Zod 校验入参
- **OpenAPI 片段**：平台根据 JSON Schema 自动渲染 OpenAPI 3.0 片段 + cURL 示例，无需手写

### 3.3 Digital Employee（数字员工）

- **= 一个 Skill + 人设层**
- 在 Skill 基础上额外拥有：名称、头像、角色定位、欢迎语、典型场景
- 在 UI 中以"人"的卡片形式呈现，但本质仍走同一调用链
- 头像统一存于对象存储（S3/MinIO），不允许外链引用
- 可理解为"被包装好的 Skill"

### 3.4 Skill 执行模型（v1）

v1 阶段 Skill 必须是**容器化 HTTP 微服务**：

- 平台以 HTTP 方式调用 Skill 暴露的 endpoint
- Skill 自带 / 内部实现 AI 推理能力，Hub 不提供模型层
- 同步调用超时 30s，超时返回失败
- v1.1 起扩展支持 Webhook 回调、模板型 Skill（仅 prompt，无独立服务）

### 3.5 调用契约

- 统一 OpenAPI 3.0 规范
- 输入参数按 Skill 声明的 JSON Schema 校验（Zod 运行期校验）
- v1 仅同步调用；v1.1 引入异步任务（task_id + 轮询）

## 4. 功能清单

### 4.1 浏览与发现

- **F1 首页** —— 数字员工与 Skill 入口分流、推荐位、最新上架、热门调用
- **F2 列表页** —— 多维筛选（分类/标签/状态/更新时间）、关键词搜索、排序（最新/最热/评分）
- **F3 详情页** —— 数字员工展示人设卡片；Skill 展示 SKILL.md 渲染、版本、调用说明、统计、反馈
- **F4 分类管理** —— 树形分类，Hub Owner 可维护

### 4.2 发布与生命周期

- **F5 Skill 创建向导** —— 填写基础信息 → 编写/上传 SKILL.md → 声明输入/输出 Schema（JSON Schema）→ 提交审核
- **F6 数字员工创建** —— 在 Skill 基础上叠加人设（名字、头像、欢迎语、角色说明）
- **F7 版本管理** —— 每次发布生成版本号（SemVer），支持查看历史与回滚
- **F8 上下架** —— 贡献者可下架自己的（`POST /api/studio/skills/:id/offline` → 状态 Offline）；**Hub Owner 可强制下架任意 Skill（留原因）**；Reviewer 不能直接下架已上架 Skill，只能打回修改

### 4.3 审核流

- **F9 审核队列** —— Reviewer 视角查看待审 Skill
- **F10 审核操作** —— 通过 / 驳回（带意见）/ 打回修改
- **F11 审核历史** —— 每个 Skill 完整记录审核人、时间、意见

### 4.4 在线试用与 API 调用

- **F12 试运行控制台** —— 详情页内嵌，按 JSON Schema 动态生成表单，提交后展示原始输出
- **F13 API 文档** —— 详情页**自动从 JSON Schema 渲染** OpenAPI 3.0 片段 + cURL 示例
- **F14 API Key** —— 用户生成自己的 Key（按 Key 维度限流与审计，v1 默认 60 次/分钟/Key）
- **F15 异步任务** —— v1.1 引入：长任务返回 task_id，前端轮询结果

### 4.5 反馈与社区

- **F16 评分** —— 1-5 星
- **F17 评论** —— 富文本评论，支持**一层回复**（不嵌套）
- **F18 收藏** —— 我的收藏列表
- **F19 点赞** —— 详情页点赞计数

### 4.6 数据与统计

- **F20 技能统计** —— 每个 Skill 详情页显示：累计调用、近 7 天调用、成功率、平均耗时
- **F21 贡献者统计** —— 我发布的、上架数、累计调用
- **F22 Hub 大盘** —— Owner 视角：总 Skill 数、调用趋势、Top 10、活跃贡献者

### 4.7 管理后台

- **F23 用户与角色** —— 增删改查、**邀请注册**、角色分配
- **F24 分类管理** —— 树形增删改
- **F25 审计日志** —— 关键操作留痕（可检索、可导出）
- **F26 强制下架** —— Owner 可对任意 Skill 强制下架并留原因

## 6. 用户流程（关键路径）

### 6.1 Skill 提交流程

```
贡献者点击"发布 Skill"
  → 填写基础信息（名称/分类/标签）
  → 编写 SKILL.md（在线编辑器 + 实时预览，按 WorkBuddy SKILL.md 规范）
  → 声明输入/输出 Schema（JSON Schema）
  → 选择是否绑定为数字员工（是：补人设信息；否：跳过）
  → 提交审核
    → 状态：待审核 (Pending)
  → Reviewer 在审核队列处理
    → 通过 → 状态：已上架 (Approved)
    → 驳回 → 状态：已驳回 (Rejected)（带意见，贡献者修改后可重新提交）
    → 打回修改 → 状态：草稿 (Draft)（带意见，贡献者改完再次提交）
```

### 6.2 用户调用流程

```
普通用户进入详情页
  → 试运行（页面表单，按 JSON Schema 动态生成）
  → 满意后复制 OpenAPI/cURL 示例
  → 在个人中心生成 API Key
  → 业务系统按文档调用（Authorization: Bearer <API_KEY>）
  → 调用记录进入 CallLog → 聚合到详情页统计
```

### 6.3 Skill 重新上架流程

```
贡献者 / Owner 将已下架的 Skill 重新发布
  → 进入我的工作台 → 选择 Offline 状态的 Skill
  → 点击"重新发布"
  → 可选：编辑新版 SKILL.md / Schema（生成新版本号，SemVer 自增 patch）
  → 提交审核
    → 状态：待审核 (Pending)
  → Reviewer 审核 → Approved（重新对普通用户可见）
```

### 6.4 强制下架流程（Owner）

```
Hub Owner 在管理后台/详情页点击"强制下架"
  → 弹窗填写下架原因（必填，10~500 字）
  → 确认 → 状态：已下架 (Offline)
  → 自动生成审计日志（actor/动作/原因/时间/目标）
  → 通知 Skill 归属人
```

## 8. 风险与假设

### 8.1 假设

- 公司允许内部部署，使用自托管 PostgreSQL + S3 兼容对象存储（MinIO 或自有 S3）
- v1 阶段用自建账号，SSO 在 v2 接入
- AI 推理能力由各 Skill 自带，Hub 不提供模型层
- v1 仅同步调用，不引入 Redis/异步任务
- 单 Owner 即可，v1 不要求多人 Owner
- **首个 Owner 凭据完全从 `.env` 读取**（email/name/password），seed 脚本不写死任何默认值；env 缺失时 seed 进程必须报错退出，避免静默创建弱口令账号

### 8.2 风险

| 风险                   | 等级 | 缓解                                                              |
| ---------------------- | ---- | ----------------------------------------------------------------- |
| 状态机被绕过           | 高   | 全部状态转移集中在 `lib/skill/state.ts`，API 路由与 UI 都必须调用 |
| 越权访问               | 高   | `assertCan(user, action, resource)` 中间件强制每个 API 调用       |
| 强制下架被滥用         | 中   | 仅 Owner 可操作 + 必须填原因 + 必留审计                           |
| API 被滥用             | 中   | API Key + 按 Key 限流（60/分钟/Key）+ 审计                        |
| 调用量上涨导致后端压力 | 中   | 同步限时（30s）+ 缓存 + 限流                                      |
| 用户数增长后搜索慢     | 低   | 1k 级别 PG 模糊搜索足够；1w+ 引入 ES                              |
| 自建账号被刷注册       | 中   | 注册需邀请码 / Owner 邀请激活（v1.1 增强）                        |

### 8.3 合规

- 审计日志需可导出为 CSV/JSON，保留期 ≥ 1 年
- 头像与 SKILL.md 附件统一存对象存储，公开访问走签名 URL
- 调用日志含 user/apiKey/skill/version/status/latency，便于事后追责
